DEVIL'S ADVOCATE CRITIQUE (Binary Eval)
========================================

Target: Code critique of signup flow, RBAC system, onboarding wizard, admin panel, and webhook/API security across the Nexley AI dashboard.

  Correctness:
    tests-pass ...... FAIL — No application tests exist anywhere in the project (all .test.ts files are in node_modules only). Cannot verify correctness.
                      Fix: Add at minimum integration tests for the API routes (signup, onboarding, webhooks). Use vitest or jest with MSW for API mocking.

    logic-correct ... FAIL — CRITICAL: proxy.ts defines the middleware RBAC logic (auth redirects, role blocking) but is NEVER imported. There is no middleware.ts file at the project root. The entire middleware RBAC layer is dead code — no auth redirects, no role-based page blocking happens at the middleware level. All "protected" pages are accessible to unauthenticated users at the network level (server components individually call rbac-guard, but API routes for dashboard data like /api/onboarding do their own auth, so this is partially mitigated — but the page shells themselves render).
                      Additionally: app/api/signup/route.ts:24 — slug collision check uses .single() which returns error if >1 row matches, but the real issue is that slug generation is deterministic from business_name with no collision suffix. Two businesses named "Smith's Plumbing" will collide and the second signup will get "A business with this name already exists" — a false rejection that is a logic bug for legitimate different businesses. Compare to onboard-client/route.ts:29-36 which handles this correctly with a suffix.
                      Fix: (1) Create middleware.ts that imports and re-exports the proxy function. (2) Add collision-resistant slug generation to signup/route.ts matching the pattern in onboard-client/route.ts.

    edge-cases ...... FAIL — app/api/signup/route.ts:55 — agentApiKey generation uses crypto.getRandomValues(new Uint8Array(24)) producing a 48-hex-char key. No check that the generated key is unique in agent_config. While collision probability is astronomically low, the insert has no unique constraint handling.
                      app/api/signup/route.ts:84-108 — If auth.admin.createUser succeeds but generateLink or sendSystemEmail fails, the user is created with email_confirm:false and no magic link is sent. The API returns { success: true, trial: true } — the user sees "Check your email!" but never receives the email. No rollback or retry.
                      app/(dashboard)/onboarding/page.tsx:64-69 — saveStep() swallows PATCH errors silently (the catch clause is missing; the finally block just sets saving=false). If the server returns an error, the user thinks they saved but didn't.
                      Fix: (1) Handle createUser/generateLink/sendEmail failure chain — if email fails, at minimum return an error or flag for retry. (2) In onboarding page, check the response status from the PATCH call and show an error if it fails.

  Security:
    no-secrets ...... PASS — No hardcoded secrets found. All sensitive values come from process.env. API keys are generated at runtime.

    input-validated . FAIL — app/api/signup/route.ts:10-11 — req.json() is parsed with no schema validation. The email field is only checked for truthiness, not format. Fields like phone, location, services, personality are passed directly to Supabase insert with no validation or sanitization. A malicious payload could include extra fields or malformed data.
                      app/api/onboarding/route.ts:63 — The PATCH body is destructured as { step, data } with no validation that step is a number or that data matches expected shape. Arbitrary keys in data are spread into Supabase updates.
                      app/api/webhook/chat/route.ts:28 — widget_key is used to look up config via .eq('booking_link', widget_key) but booking_link is a field repurposed as widget key. The widget_key value is never validated for format.
                      Fix: Add Zod schemas to validate all API request bodies. At minimum: signup (email format, string lengths, enum for plan/personality/industry), onboarding (step as 1|2|3, typed data per step), chat webhook (widget_key format).

    no-injection .... FAIL — app/api/webhook/meta/route.ts:93 — `recipientId` (from untrusted Meta webhook payload) is interpolated directly into a PostgREST filter string: `.or(\`metadata->facebook_page_id.eq.${recipientId},metadata->instagram_user_id.eq.${recipientId}\`)`. If recipientId contains PostgREST operators (commas, parentheses, dots), this could alter the query semantics. This is a PostgREST filter injection vector.
                      app/api/webhook/meta/route.ts:54 — When META_APP_SECRET is not configured, signature validation is completely skipped and the webhook processes arbitrary JSON. The log says "signature validation disabled" but the code proceeds to process the payload. This allows anyone to inject fake messages.
                      Fix: (1) Sanitize recipientId before interpolation, or use separate .eq() calls instead of string interpolation in .or(). (2) When META_APP_SECRET is missing, reject the request (fail closed) instead of proceeding without validation.

    auth-enforced ... FAIL — CRITICAL: proxy.ts middleware is dead code (never imported, no middleware.ts exists). This means:
                      - Unauthenticated users can load any dashboard page shell (though API calls from those pages will fail with 401).
                      - Role-based page blocking defined in proxy.ts lines 67-78 never executes.
                      - The MEMBER_BLOCKED list in proxy.ts:6 (['/settings', '/money', '/agent-logs', '/integrations']) differs from MEMBER_BLOCKED_PAGES in lib/rbac.ts:41 (['/settings', '/integrations']) — but since neither executes at middleware level, both are moot.
                      Server components that call requireRole/requireAccess (like admin/page.tsx:30) ARE protected, but client components and pages without those guards are not.
                      app/api/signup/route.ts — The signup endpoint has no rate limiting. An attacker could mass-create clients and auth users, filling the database and Supabase auth table.
                      Fix: (1) Create middleware.ts at project root that imports { proxy } from './proxy' and exports it as middleware with the matcher config. (2) Add rate limiting to the signup endpoint (e.g., Vercel Edge rate limiting or IP-based throttling). (3) Reconcile the two MEMBER_BLOCKED lists.

                      Fix: (1) Wire up proxy.ts via middleware.ts. (2) Either use buildWeeklyReportHtml in the cron route or delete it. (3) Move PERSONALITIES and INDUSTRIES to a shared constants file.

    no-placeholders . PASS — No TODO/FIXME/HACK comments found in app/ or lib/ directories.

    error-handling .. FAIL — app/(dashboard)/onboarding/page.tsx:63-71 — saveStep() has no error handling for the fetch response. The try block calls fetch but never checks res.ok. Errors are silently swallowed and the user advances to the next step thinking their data was saved.
                      app/api/signup/route.ts:57-80 — The agent_config insert result is never checked for errors. If it fails, the client row exists but has no agent_config — an orphaned client with no AI agent configuration.
                      app/api/webhook/meta/route.ts:205 — Errors return 200 status ("Must still return 200 to prevent Meta from disabling webhook") which is correct for Meta, but the error is only logged to console — no alerting mechanism.
                      Fix: (1) Check response.ok in onboarding saveStep and show error to user. (2) Check agent_config insert error in signup route and roll back client creation if it fails. (3) Add error alerting for webhook failures (e.g., Sentry, or at least a webhook_log entry with error status).

    no-code-smell .. FAIL — proxy.ts duplicates RBAC constants from lib/rbac.ts with DIFFERENT values (proxy.ts MEMBER_BLOCKED includes '/money' and '/agent-logs' which rbac.ts MEMBER_BLOCKED_PAGES does not). This is a data clump / inconsistency that would cause different behavior depending on which enforcement path runs.
                      app/api/signup/route.ts mixes concerns: it handles client creation, agent config setup, auth user creation, magic link generation, email sending, AND Stripe checkout — all in one 131-line function. This is a god function.
                      Fix: (1) Consolidate RBAC constants to a single source of truth in lib/rbac.ts and import from there in proxy.ts. (2) Extract signup sub-operations into helper functions (createClient, createAgentConfig, setupTrialAuth, createCheckoutSession).

  Performance:
    no-obvious-perf . FAIL — app/(dashboard)/admin/page.tsx:40-42 — Three separate count queries run sequentially (totalContacts, totalCalls, totalOpportunities). These are independent and could be parallelized with Promise.all().
                      app/api/cron/weekly-report/route.ts:199 — N+1 query pattern: for each client, a separate query fetches call_logs. With 100 clients, this is 100+ queries.
                      app/api/cron/follow-ups/route.ts:44-48 — Same N+1 pattern: for each stale contact, fetches agent_config separately.
                      Fix: (1) Use Promise.all() for the three admin count queries. (2) For cron routes, batch-fetch all call_logs and agent_configs in single queries, then group in-memory.

  Consistency:
    types-consistent .. FAIL — lib/rbac.ts defines UserRole as 'super_admin' | 'owner' | 'member', but proxy.ts:65 reads role from user.app_metadata?.role with a fallback to 'member' as a plain string — no type assertion. The two files maintain parallel but divergent role-checking logic.
                        app/api/onboarding/route.ts:4 imports getSupabase from '@/lib/supabase' (the service role client), while route.ts:3 imports createClient from '@/lib/supabase/server' (the user-scoped client). Both are used in the same file for different purposes — the service client is used for data mutations (bypassing RLS) while the user client is for auth. This pattern is intentional but undocumented and inconsistent with how other routes handle it.
                        Fix: (1) Have proxy.ts import UserRole from lib/rbac.ts and use it. (2) Document the dual-client pattern in a comment or CLAUDE.md.

    naming-matches ... PASS — Naming conventions are consistent: snake_case for database fields, camelCase for JS variables, kebab-case for URL slugs. Component files use PascalCase.

    patterns-followed  FAIL — Agent API routes (agent/config, agent/contacts, etc.) consistently use authenticateAgentRequest() for auth. But webhook routes use 3 different auth patterns: (a) Stripe uses manual signature verification, (b) Twilio uses x-twilio-webhook-secret header comparison, (c) Meta uses X-Hub-Signature-256 HMAC but falls open when secret is missing, (d) ElevenLabs uses x-webhook-secret header. The Stripe pattern is the most robust (HMAC + replay protection + timing-safe compare). The others use plain string comparison which is potentially timing-attackable.
                       Fix: Standardize webhook auth: (1) All webhooks should fail closed when secrets are not configured. (2) Use timing-safe comparison for all secret comparisons (crypto.timingSafeEqual), not just Stripe.

  Architecture:
    boundaries-respected .. FAIL — proxy.ts defines middleware-level RBAC enforcement but is disconnected from the actual application (no middleware.ts imports it). This creates a false architectural boundary — the code suggests middleware enforcement exists, but it does not. Developers reading proxy.ts would believe routes are protected when they are not.
                            app/api/signup/route.ts:6-7 creates its own Supabase client directly instead of using the shared getSupabase() from lib/supabase. Three different patterns exist for getting a service-role Supabase client: (a) getSupabase() singleton, (b) inline createClient() in route files, (c) createClient in lib/api-auth.ts. This violates the service boundary — there should be one way to get the admin client.
                            Fix: (1) Wire up middleware.ts. (2) Consolidate all service-role Supabase client creation to use getSupabase() from lib/supabase.

    no-hacky-shortcuts .... FAIL — The Meta webhook (meta/route.ts:61-63) falls OPEN when META_APP_SECRET is not configured: it skips signature validation entirely and processes untrusted payloads. This is a "skip the security check" shortcut for development that is dangerous in production.
                            The chat widget webhook (chat/route.ts:58-65) allows all origins when no allowed_origins are configured — another open-by-default pattern that should be closed-by-default.
                            app/api/signup/route.ts slug generation silently fails on collision instead of generating a unique slug — a shortcut that pushes the problem to the user ("A business with this name already exists").
                            Fix: (1) Meta webhook: fail closed when secret is missing. (2) Chat widget: reject requests when no origins configured, or require explicit opt-in to allow-all. (3) Signup: generate collision-resistant slugs.
